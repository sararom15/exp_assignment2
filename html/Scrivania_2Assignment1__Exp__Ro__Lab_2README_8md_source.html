1 # Assignment 1 - Experimental Robotics Laboratory
2 # Behavioral Architecture
3 Assuming a Robot, simulating a pet, that interacts with a human and moves in a discrete 2D environment.
4 The robot may have three behaviors : normal (in which it moves randomly), sleep (in which it gets the home position, sleeps for a time and returns in normal behavior) and play (in which it goes in person location, waits for a poiting gesture, goes in the pointed location, comes back to the person and waits for the next pointing gestures. After some time it returns to the normal behavior).
5 The human can interact by pointing gestures and speech.

15 ### Commander node
16 The node simulates user action.
17 Firstly it chooses randomly what the robot must perform: sleep or play. The command (std_msgs/String) is published on a Topic ("/Command").
18 In case the play behavior is selected, the node generates randomly the location of the user and the Pointing Gesture.
19 These poses (geometry_msgs/Point) are published on two topics ("/PersonPosition" and "/PointingGesture").
20 
21 ### State machine node
22 The node implements a state machine in which the three possible behaviors are defined.
23 
24 <p align="center">
25 <img src="https://github.com/sararom15/Assignment1_Exp_Ro_Lab/blob/main/Images/state%20machine.png">

27 
28 The state machine starts with a Normal behavior state, and it can transit to sleep state or to play state, or it can keep a normal state.
29 The command to switch to another state is received by "Commander" node by the topic "/Command".
30 When it is in a Play state or in a Sleep state, it can just go back in the normal state.
31 The ROS message geometry_msgs/Point is used to rapresent the 2D robot's position (leaving z component always equal to 0).
32 The target position belongs exclusively to the map (11x11 grid).
33 In the normal state, they are generated by "GenerateRandomPosition" function; in the play state, the person position and pointing gesture are received by "Commander" node through two topics; while in the sleep state the home position is defined (1,1).
34 For each state, the target positions are sent to the "display" node, publishing them on "NewTargetPosition" topic.
35 When this happens, the system waits for a default time (4 seconds) until the robot reaches the desired pose. This means that further targets has not been accepted while the robot is moving.
36 
37 ### Display node
38 The node is a simple simulator; it subscribes to a "newTargetPosition" topic and wait for a message.
39 When the target is received, it prints on terminal what the robot is doing and the behavior state.

69 ## Working Hypothesis
70 The assumptions are:
71 1) When it starts, the normal state preempts other states.
72 2) From the normal behavior, the robot can play or sleep or stay in normal state yet.
73 3) The command to go to sleep or play is received by the Commander node.
74 4) From the play state, as well as from the sleep state, the robot can only go to the normal state.
75 5) The random position in the normal state is defined by "GenerateRandomPosition" function.
76 6) The home position is fixed (1,1,0).
77 7) The person location and the pointing gesture are received by the Commander node.
78 8) During the execution of one state, after giving a target, the robot waits for a while (predefined time).

84 ## System's Limitations
85 The user actions is randomly defined by the system, and there exists neither a real relationship with the user nor a real pointing gesture.
86 The robot has not a velocity to get him desired target position, and there is not a real simulator to view if the target is achieved.
